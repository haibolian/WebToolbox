# WebToolbox 技术栈设计（前端版）

**日期：** 2026-03-04  
**状态：** 已确认（用户认可）

## 1. 背景与目标
- 项目当前为空仓库。
- 产品方向：Web 工具箱，后续可持续增加工具（如图片编辑）。
- 当前约束：纯前端、在线使用（暂不要求 PWA）。
- 优先级：
  - A：快速上线与开发效率
  - B：长期扩展性

## 2. 备选方案与取舍
### 方案 A（推荐）：React + TypeScript + Vite
- 优点：生态成熟、组件与编辑器相关库丰富、招聘与协作成本低、Vite 启动与构建快。
- 缺点：样板代码略多于 Vue/Svelte。

### 方案 B：Vue 3 + TypeScript + Vite
- 优点：上手快、SFC 开发体验好。
- 缺点：在复杂编辑器场景下，团队与生态选择面通常略窄。

### 方案 C：Svelte + Vite
- 优点：语法简洁、性能好。
- 缺点：长期扩展与团队协作生态相对弱。

## 3. 最终选型
- 核心：React + TypeScript + Vite（SPA）
- 路由：React Router
- 状态：Zustand
- UI：Tailwind CSS
- 测试：Vitest + React Testing Library + Playwright
- 包管理：pnpm

## 4. 架构设计（v1）
### 4.1 目录分层
- `src/app`：应用入口、路由、全局布局
- `src/tools`：各个工具模块（按工具隔离）
- `src/shared`：共享 UI/工具函数

### 4.2 工具扩展模型
- 每个工具独立目录（例如 `src/tools/image-editor`），包含页面、组件、状态。
- 工具入口通过统一注册表管理，避免未来路由与导航散落。

### 4.3 数据流与状态
- 全局 UI 状态（如主题、侧边栏）由 Zustand 管理。
- 工具内部状态优先本地化；仅跨页面共享时才提升到全局。

## 5. 错误处理与质量策略
- 路由层提供 404 页面。
- 页面级别保留错误边界扩展位。
- 采用 TDD 节奏建立基础行为测试：首页渲染、导航跳转、404 行为。

## 6. 测试与验收
- 单测：Vitest + RTL，覆盖核心路由与基础交互。
- E2E：Playwright，覆盖首页到图片编辑页导航流程。
- 构建：`pnpm build` 作为发布前门禁。

## 7. 非目标（当前阶段）
- 不引入后端服务。
- 不做离线缓存/PWA。
- 不做复杂插件系统，仅保留演进空间。
